## 一、服务端代码 Review

### 总体评价
整体代码质量不错，结构清晰，功能完整，符合腾讯云SCF开发规范。👍

**优点：**
1. 三级 API Key 权限分离设计合理（admin/client 区分，管理接口需要 admin 权限）
2. 数据库迁移处理做得好（自动检测并升级表结构，向后兼容旧数据）
3. 动态权限配置设计优雅，权限存在数据库中，可以通过接口动态修改，不需要改代码重启
4. 有效期累计逻辑正确，同一个设备多个激活码会自动累加有效期
5. 错误处理完善，各个接口都有参数校验，异常捕获
6. 文档齐全，README 写得很清楚，部署步骤详细

### 发现的问题 & 修复建议

| 序号 | 问题 | 严重程度 | 修复建议 |
|------|------|----------|----------|
| 1 | 缺少 `base64` 模块 import，当请求体被base64编码时会抛出 `NameError` | 🐛 **严重** | 在文件开头添加 `import base64` |
| 2 | 401 响应调试信息返回了完整的 `client_api_keys` 明文，存在信息泄露 | ⚠️ 安全 | 删除 `client_api_keys` 明文返回，只保留计数 |
| 3 | `/auth/generate` 接口没有限制 `count` 参数最大值，可能被一次性生成大量激活码打爆数据库 | ⚠️ 重要 | 添加限制 `count = min(max(1, count), 100)`，一次最多100个 |
| 4 | 频率限制使用内存存储，多实例无法全局限流，且不会清理过期记录导致内存增长 | 优化 | 添加过期记录清理，或者说明限制接受 |
| 5 | 调试日志打印完整API Key，日志泄露风险 | 优化 | 只打印掩码后的key |

### 三级 API Key 实现评估
**结论：设计合理，实现正确。**

三级是指：
1. **Admin API Key** → 可以访问所有管理接口（生成/删除/更新）✅
2. **Client API Key** → 只能访问用户端接口（验证、查询设备信息）✅
3. **激活码** → 用户层面的授权，绑定设备 ✅

### 数据库结构评估
**结论：设计良好，支持向前兼容。**

- `activation_codes` 表结构完整，正确
- `package_permissions` 按 `(client_type, package_type)` 存储权限配置，JSON格式灵活扩展，支持动态修改，设计非常好👍

## 四、运营需求文档

### 激活码销售流程

```
用户下单支付 → 系统生成激活码 → 自动发货给用户 → 用户在客户端激活 → 服务端验证生效 → 后期运营维护
```

### 核心统计指标

1. **生成量**：每日按客户端/套餐/时长统计
2. **激活量**：每日激活数、累计用户数、激活率
3. **到期量**：每日到期数、即将到期预警（7天/30天）
4. **营收**：每日营收、套餐占比、累计营收
5. **活跃**：每日/每周/每月活跃设备数

### 日报表示例

**日期：** YYYY-MM-DD

| 指标 | 今日 | 昨日 | 环比 | 累计 |
|------|------|------|------|------|
| 生成激活码 | | | | |
| 成功激活 | | | | |
| 到期失效 | | | | |
| 新增用户（去重） | | | | |
| 营收金额 | | | | |

### 到期预警

- 统计未来7天/30天内即将到期的激活码数量，按日期展示
- 方便运营做续费提醒营销

### 运营工具需求

| 工具功能 | 优先级 |
|---------|--------|
| 订单管理界面 | 高 |
| 批量导出激活码 | 高 |
| 数据报表看板 | 中 |
| 续费提醒推送 | 中 |

### 建议订单表结构

```sql
CREATE TABLE orders (
    order_id TEXT PRIMARY KEY,
    client_type TEXT NOT NULL,
    package_type TEXT NOT NULL,
    duration INTEGER NOT NULL,
    auth_code TEXT NOT NULL,
    amount DECIMAL(10,2) NOT NULL,
```

### MVP 阶段建议

首发可以先简化：
- 自动发货：电商平台自动回复，生成后直接发给用户
